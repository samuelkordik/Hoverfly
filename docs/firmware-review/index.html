1. Dual-Z wiring — two motors, parallel, on one driver LOW

The "Dual Z-axis" mod is two Z steppers + lead screws driven from the single Z output via the Creality kit's parallel Y-splitter cable. The firmware is correctly set for one Z driver (Z2_DRIVER_TYPE stays commented): the SKR Mini E3 V2.0 has only four drivers (X, Y, Z, E0) and no spare socket, so a second independent Z driver isn't possible on this board. This works — the notes below are optional.

Parallel current — the one thing to watch. Both motors share the one TMC2209, so they split its current. Stock Z current is 580 mA (Z_CURRENT, Configuration_adv.h:2777). If Z ever lacks torque or skips, raise it via Z_CURRENT (reflash) or live with the SET_ZCURRENT.gcode SD macro (M906 Z + M500, UART mode, no reflash). The driver's set current is the total it sources for both motors — it divides ~50/50 on average but shifts with each motor's load/back-EMF, so budget against the driver's ≈1.4 A RMS ceiling (total), not a per-motor figure. Keep it ≤ ~1000 mA and check the driver isn't running hot. Z moves slowly, so this is rarely a problem in practice.
Series is cleaner, but it's a hardware swap. Wiring the two motors in series gives both full, synchronized current within the driver's rating (Z's low speed makes the doubled inductance irrelevant) — the conventional choice for two motors on one driver. But the stock Creality cable is the parallel one, so series needs a different splitter cable. Optional, not urgent.
No auto gantry-align. With one Z driver you can't use Z_STEPPER_AUTO_ALIGN / G34 — square the gantry manually (power off, hand-turn both lead screws to equal height). This stays true even under the planned Klipper move (still one Z driver). Also update the README mod list to say "two Z motors, parallel-wired to the single driver."

Fan wiring & cooling — dual 5015 blowers

Three fans, two controllable headers on the SKR Mini E3 V2.0 (FAN0 and FAN1, both 24 V PWM). Your current wiring is already the right scheme:

FanConnect toFirmwareBehaviour
Hotend / heatbreakFAN1E0_AUTO_FAN_PIN FAN1_PINAuto-on when the hotend is hot (EXTRUDER_AUTO_FAN_TEMPERATURE 50, speed 255). Cools the heatbreak only when needed.
Parts cooling — dual 5015 blowersFAN0 → Sprite PCB part-fan connector (two blowers spliced)default M106 PWMSlicer-controlled. Two 5015s ≈ 0.2 A total — well within the FAN0 header.
Mainboard / controllerPWR (always-on)USE_CONTROLLER_FAN offHard-wired to power; runs continuously. Correct as-is — it isn't on a controllable header, so leave the define off.
One firmware add for the blowers. 5015 blowers often won't start spinning at low slicer PWM. Give them a kickstart pulse and a floor: 
// Configuration_adv.h
#define FAN_KICKSTART_TIME 150   // (ms) full-speed pulse on start
#define FAN_MIN_PWM 75           // out of 255 (~29%), not a percent — floor so they don't stall
Then the slicer's 0%→100% ramp won't leave them stalled at low speeds.
FAN_MIN_PWM only rescales non-zero speeds. A commanded 0 is always fully off (Marlin's own comment: "Value 0 always turns off the fan."), so your layer-1 0% fan still means no airflow — PLA bed adhesion is unaffected. The ~29% floor applies only when the slicer asks for a low but non-zero speed.
PLA duct caveat. The Taurus V5 duct is printed in PLA, which softens near ~60 °C — right next to the hotend it can sag over long/hot prints. Keep an eye on it and reprint in PETG/ABS/ASA if it deforms. Cooling itself is great — dual blowers move a lot of air, so keep the layer-1 fan low for bed adhesion (the Orca profiles already ramp 0%→100% by layer 3).

2. Linear Advance K is enabled but set to 0.0 HIGH

LIN_ADVANCE is compiled in, but ADVANCE_K 0.0 means it does nothing. Linear Advance pays off most on a light direct-drive setup like the Sprite Pro: it pressure-compensates the extruder so corners, line starts/stops and seams stay crisp instead of bulging. The default 0.0 leaves that on the table.

S-curve + Linear Advance is fine on this Marlin (not a conflict). S_CURVE_ACCELERATION (Configuration.h:1293) and LIN_ADVANCE are both on, and on Marlin 2.1.2.7 that is supported — the old sanity-check #error that rejected the pair was removed. Current Marlin guards the LA/S-curve math with #if ANY(S_CURVE_ACCELERATION, LIN_ADVANCE) (planner.cpp, planner.h) and compiles both together. No EXPERIMENTAL_SCURVE is needed — in fact defining it now breaks the build (inc/Changes.h: "EXPERIMENTAL_SCURVE is no longer needed and should be removed"), and it is not set here. The build is not secretly broken. Disabling S-curve is now an optional preference, not a fix: some find Linear Advance pressure timing crispest with it off, since junction deviation already smooths corners. 
// Configuration.h
//#define S_CURVE_ACCELERATION   // optional: try LA without S-curve
Refs (historical, now resolved): Marlin #22547, #14728.

Recommended: calibrate K. Direct-drive PLA on a Sprite typically lands around 0.02–0.10 (much lower than Bowden’s ~0.4–0.8). Don’t guess — measure it:

  • Use Marlin’s K-factor test pattern, or (easier) the Orca Slicer “Pressure advance” calibration test — see the Calibration Guide §6.

How to apply (live, no reflash):

M900 K0.04      ; set Linear Advance K (use your measured value)
M500            ; save to EEPROM
M900            ; report current K

To bake the default into firmware instead, set in Configuration_adv.h: #define ADVANCE_K 0.04.

If you also run pressure advance in the slicer, pick one — don’t apply both. Either tune Marlin M900 K and leave slicer PA off, or disable LIN_ADVANCE influence and let Orca do it. Tuning it in firmware is the more portable choice.

3. Enable BABYSTEP_ZPROBE_OFFSET MED

Babystepping is enabled (with double-click entry), which is great, but BABYSTEP_ZPROBE_OFFSET is off. With it on, the live Z babysteps you make during the first layer directly adjust the probe Z-offset, so you can dial the first layer by feel and then M500 to make it permanent — exactly the workflow you want with a CRTouch.

// Configuration_adv.h
#define BABYSTEP_ZPROBE_OFFSET

After flashing, adjust during a print (double-click the wheel → Z babystep), then save with M500. See Calibration Guide §4.

4. Bed temperature: bang-bang vs PID MED

PIDTEMPBED is disabled, so the bed uses bang-bang (full-on/full-off) control. For PLA this is genuinely fine and reduces MOSFET cycling. Enable bed PID only if you want tighter bed-temp stability (e.g. for finicky first layers or PETP later). If you enable it, autotune rather than trusting the fallback constants:

// Configuration.h — then rebuild
#define PIDTEMPBED
; live autotune (no reflash needed to test):
M303 E-1 S60 C8 U1   ; bed (E-1), 60C target, 8 cycles, apply result
M500                 ; save

Recommendation: leave bang-bang for now; revisit only if bed temp wanders.

5. PROBING_MARGIN is very large (50 mm) MED

With X/Y_BED_SIZE 210 and PROBING_MARGIN 50, the mesh only samples the central ~110 × 110 mm of the bed — the outer ~50 mm ring is never probed and is extrapolated. Combined with the large probe offset ({-30,-40}) this was likely chosen to keep the probe physically on the bed, but 50 mm is conservative. Reducing it meshes more of your actual print area.

#define PROBING_MARGIN 25   // was 50 — Marlin auto-clamps the grid to X∈[25,180] Y∈[25,170]; stays on bed
Marlin auto-clamps the probe grid so the pin always stays on the bed — no trial-and-error off a hot bed needed. With offset {-30,-40} and PROBING_MARGIN 25, probe.h's _min/_max bounds give the sampled region X ∈ [25, 180], Y ∈ [25, 170] (max_x = MIN(210−25, 210−30) = 180, max_y = MIN(210−25, 210−40) = 170). The far edges are pulled in by the offset, not the margin; the margin only governs how close to the front/left near edges the pin lands. The probe never deploys off the bed at 25 (it stays on even down to ~10), so 25 is a conservative, verified-safe choice.

6. Bed mesh density 4×4 → 5×5 LOW

A 4×4 grid is coarse for a 210 mm bed. 5×5 (25 points) better captures a warped or crowned bed with little added probe time. Bilinear interpolation between denser points = a more accurate first layer.

#define GRID_MAX_POINTS_X 5   // GRID_MAX_POINTS_Y inherits this
Probe the mesh at print temperature. PREHEAT_BEFORE_LEVELING is on (good), but LEVELING_BED_TEMP is currently set to 50 (Configuration.h:1936) — below your ~60 °C PLA bed, so the mesh is probed ~10 °C cold. An aluminum bed warps differently when hot, so a mesh probed cold misses warp that's present while printing. Set it to your usual bed temp, or probe fresh in the slicer start G-code after the bed soaks: #define LEVELING_BED_TEMP 60 — or M190 S60 → G4 S300 (soak) → G28 → G29 → M500. (An M48 P10 probe-repeatability check — target σ < 0.02 mm — is worthwhile too, but it only reports over serial, so it waits for the Klipper console.) See Calibration Guide §9.

7. Input Shaping on an STM32F103 MED

Input Shaping (INPUT_SHAPING_X/Y) cancels ringing/ghosting and lets you run higher acceleration without artifacts — a real quality win. The catch: the SKR Mini E3 V2.0 is an STM32F103RC (72 MHz, 48 KB SRAM). Marlin IS is supported but memory- and CPU-hungry; on this MCU the build can overflow or the step rate can be limited.

  • Try it: enable INPUT_SHAPING_X / INPUT_SHAPING_Y, set the shaper frequencies from a ringing-tower measurement (Calibration Guide §8), tune live with M593 X F<freq> / M593 Y F<freq>, then M500.
  • If it won’t fit / build errors on flash overflow: drop it. The pragmatic alternative on this board is to keep acceleration modest, or move to Klipper (the planned migration — input shaping and resonance measurement are first-class there).
// Configuration_adv.h
#define INPUT_SHAPING_X
#define INPUT_SHAPING_Y
// after flashing, live-tune:
//   M593 X F40 ; M593 Y F38 ; M500   (use YOUR measured frequencies)
Making it fit on 48 KB SRAM. IS allocates a step buffer in SRAM that grows with steps/mm and feedrate; bound it with SHAPING_MIN_FREQ 20.0 (and optionally SHAPING_MAX_STEPRATE). After enabling, build and read the linker summary before flashing — if it reports "region RAM overflowed", free SRAM (drop unused LCD languages, keep SHAPING_MENU off) or abandon IS. This printer is low-risk (≈80 steps/mm XY, modest accel), but it's a real ceiling on the F103. 
// Configuration_adv.h
#define SHAPING_MIN_FREQ 20.0   // caps the SRAM step buffer
// keep //#define SHAPING_MENU commented to save RAM
Tuning order: measure frequency from the ringing tower → set M593 X F.. Y F.., then tune the damping factor M593 X D.. Y D.. (zeta, ~0.15 start, step 0.05). Re-tune Linear Advance K after Input Shaping — the IS sweep runs with LA off, so K is finalized last (mechanics → IS → LA K). See Calibration Guide §8.

8. Motion limits, junction deviation, S-curve LOW

SettingCurrentComment
DEFAULT_ACCELERATION500Safe/conservative. The Sprite is light; you can likely raise print accel to 1000–2000 after a ringing test. Tune empirically.
JUNCTION_DEVIATION_MM0.08Good general value. Lower (0.05) = crisper corners/slower; higher (0.10–0.13) = faster/rounder. Leave at 0.08 unless chasing speed.
S_CURVE_ACCELERATIONonFine with LIN_ADVANCE on this Marlin version (see §2); disabling it is optional preference, not a fix. Junction deviation already handles corner smoothing.
DEFAULT_MAX_FEEDRATE Z5 mm/sConservative for a dual-Z leadscrew; fine. Don’t raise much.
DEFAULT_MAX_ACCELERATION E5000Plenty for direct drive. No change.

Raise limits only with a test (see Calibration Guide §7); the firmware ceiling should be ≥ whatever the slicer asks for.

Gotcha — E max feedrate caps retraction speed. DEFAULT_MAX_FEEDRATE E is 25 mm/s, so any slicer retraction speed above 25 is silently clamped to 25. The common "40 mm/s" retraction (and the older numbers in the slicer/calibration guides) never actually happens on this machine. Either set slicer retraction speed to ≤25 mm/s to match reality, or raise the E feedrate ceiling (M203 E50 + M500, or the last value of DEFAULT_MAX_FEEDRATE) if you genuinely want faster retracts. A light Sprite direct drive rarely needs more than ~25.

9. Smaller items & quality-of-life LOW

  • ALLOW_LOW_EJERK — DEFAULT_EJERK is 5.0 while LIN_ADVANCE is on. With CLASSIC_JERK disabled you're on junction deviation, so Marlin's DEFAULT_EJERK >= 10 check is compiled out entirely (SanityCheck.h guards it with NONE(HAS_JUNCTION_DEVIATION, ALLOW_LOW_EJERK)) — DEFAULT_EJERK 5.0 is harmless and ALLOW_LOW_EJERK is not needed here. It would only matter if you re-enabled classic jerk.
  • G26_MESH_VALIDATION — currently commented out. Enable it to print a single-layer pattern across the whole bed (G26) that visually confirms your mesh yields a flat first layer everywhere — the companion to the M48 probe-repeatability check.
  • PROBING_HEATERS_OFF — turning heaters off during probing reduces electrical noise / PWM interference on the CRTouch signal and can improve mesh repeatability. Optional; enable in Configuration_adv.h if you see noisy probing.
  • POWER_LOSS_RECOVERY — off. Nice for long prints on unreliable power, but it adds SD writes and a small first-layer pause. Enable only if you need it.
  • Sensorless homing — the TMC2209s support StallGuard, but the Ender 3 already has physical X/Y endstops and the SKR Mini E3 V2.0 needs DIAG jumpers. Not worth it here. No change.
  • TEMP_SENSOR_0 = 1 — generic 100k thermistor. Correct for most Creality/Sprite hotends; if temps read oddly, confirm the Sprite Pro’s thermistor type.
  • EDITABLE_STEPS_PER_UNIT — already enabled. Good; lets you tweak steps/mm via M92 + M500 without reflashing.
  • THERMAL_PROTECTION_CHAMBER — enabled though there’s no chamber heater. Harmless; no action.

How to apply & save changes

Two paths, depending on whether a setting needs a recompile:

PathUse forSteps
Value change (no terminal)K-factor, PID, steps/mm, mesh, Z-offset, accel, shaper freq There's no G-code console here — deliver via an LCD-menu action, an Orca calibration test, or an SD macro from Calibration_Guide/macros/ (each saves with M500; or LCD → Store Settings). M48/M503/M420 V are serial-only — skip them (defer to the Klipper console below).
RecompileFeature toggles: BABYSTEP_ZPROBE_OFFSET, PIDTEMPBED, INPUT_SHAPING_*, GRID_MAX_POINTS, PROBING_MARGIN Edit Firmware/Marlin/Configuration*.h, build (PlatformIO), flash firmware.bin via SD card, then re-run calibration and M500.
Workflow tip: make one firmware change at a time, reflash, verify, and M500 before moving on — so if something regresses you know which change did it. Keep the firmware edits on a branch (as you did for the Sprite e-steps calibration).

Future: Klipper migration (planned)

Longer-term plan: host Klipper on a wiped x86 ThinkPad (no Raspberry Pi needed — a laptop host is fully supported) and reflash the SKR Mini E3 V2.0 as the Klipper MCU; it's a Klipper reference board. Marlin-now is the priority — this is a separate project that may pair with other upgrades. Two things make it attractive here:

  • It solves the no-terminal problem. Mainsail/Fluidd give a web G-code console — you can type commands and see output, so the serial-only blockers (M48 probe repeatability, M503, mesh dumps) and PID/shaper tuning all become live.
  • Input Shaping is first-class and not constrained by the STM32F103's RAM the way Marlin IS is (§7).
